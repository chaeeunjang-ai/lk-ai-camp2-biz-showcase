# GitHub Showcase PR 제출 프롬프트

## 🎯 목표
아래 GitHub 레포에 나의 Showcase를 PR로 제출한다.
- **레포**: https://github.com/arin-ship-it/lk-ai-camp2-biz-showcase

---

## 📋 작업 순서

### 0. 사전 준비
- 사용자에게 영문 이름을 먼저 확인한다.
- `gh auth status`로 GitHub CLI 인증 상태를 확인한다. 인증이 되어있지 않을 경우, 아래 순서대로 사용자에게 단계별로 안내하고 각 단계마다 완료 여부를 확인한 뒤 다음 단계로 진행한다.
  1. `gh auth login` 실행
  2. `GitHub.com` 선택
  3. `HTTPS` 선택
  4. `Login with a web browser` 선택 후 브라우저에서 GitHub 로그인
  5. 인증 완료 후 `gh auth status`로 재확인

### 1. 레포 Clone *(최초 1회만 수행)*
해당 레포를 clone한다. 이미 clone된 레포가 존재할 경우 이 단계는 건너뛴다.

### 2. PR 및 브랜치 확인
- 같은 사용자 이름으로 이미 열려있는 PR이 있는지 먼저 확인한다.
  - **열린 PR이 있는 경우**: 해당 PR의 브랜치를 확인한다.
    - 브랜치가 로컬에 존재하는 경우: 해당 브랜치를 체크아웃하여 작업을 이어간다.
    - 브랜치가 로컬에 없는 경우: 리모트에서 `git fetch` 후 체크아웃하여 작업을 이어간다.
    - 3번으로 진행한다.
  - **열린 PR이 없는 경우**: 로컬 및 리모트에 동일한 브랜치명이 존재하는지 확인한다.
    - **브랜치가 존재하고 main에 병합되지 않은 경우**: 해당 브랜치를 체크아웃하여 작업을 이어간다.
    - **브랜치가 존재하지 않거나 이미 main에 병합된 경우**: 아래 기준으로 새 브랜치를 생성한다.
      - base 브랜치는 반드시 `main`으로 설정하며, `git pull`을 통해 `main`의 최신 상태를 반영한 뒤 브랜치를 생성한다.
      - 브랜치명 형식: `내영문이름/showcase`
      - 해당 브랜치명이 이미 존재할 경우, `내영문이름/showcase-v2`, `v3` 형식으로 버전을 올려 새 브랜치를 생성한다.
    - 3번으로 진행한다.

### 3. 파일 준비
- `submissions/내영문이름/SHOWCASE.md` 파일이 이미 존재하는지 확인한다.
  - **파일이 없는 경우**: `template/SHOWCASE.md`를 `submissions/내영문이름/` 폴더에 복사한 뒤 내용을 채운다.
  - **파일이 이미 존재하는 경우**: 기존 파일을 그대로 가져와 수정이 필요한 부분을 함께 검토하고 수정한다.

### 4. SHOWCASE.md 작성
- ① 항목을 질문할 때, ①~⑥을 한꺼번에 작성해도 된다고 안내한다.
- 사용자가 한꺼번에 작성한 경우: 전체 내용을 한 번에 파일에 반영한다.
- 사용자가 하나씩 답변하는 경우: 항목마다 답변을 받은 뒤 파일에 반영하고, 다음 항목으로 넘어간다.
- 모든 항목 작성이 완료되면, 전체 내용을 한 번 보여주고 수정할 항목이 있는지 확인한다.
  - **수정이 필요한 경우**: 해당 항목만 다시 작성하고, 완료 후 다시 전체 내용을 확인한다. 수정이 없을 때까지 반복한다.
  - **수정이 없는 경우**: 5번으로 진행한다.

| # | 항목 |
|---|------|
| ① | 해결하고 싶은 문제 |
| ② | 기존 방식의 병목 |
| ③ | 지향했던 방향성 |
| ④ | 현재 구현 단계 |
| ⑤ | 추후 과제 |
| ⑥ | 배운 점 / 소감 |

### 5. 커밋 & PR 생성
- 작업 완료 후 커밋한다.
  - **열린 PR이 없었던 경우**: 새 PR을 생성한다. PR의 base 브랜치는 반드시 `main`으로 설정한다.
  - **열린 PR이 있었던 경우**: 새 PR을 생성하지 않고 해당 브랜치에 push하여 기존 PR을 업데이트한다.
- PR 생성 또는 업데이트 완료 후, PR 링크를 제시한다.
